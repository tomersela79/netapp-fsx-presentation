Backup A: Data Lifecycle & Efficiency

Four Mechanisms. No Wasted Storage.

FSx ONTAP manages data lifecycle automatically  ·  No scripts  ·  No application changes  ·  Tiering and compression run without human input

Mechanism 1 Active data
Dedup & Compression
Data arrives via SnapMirror onto the hot SSD tier; footprint reduced automatically.
Compression runs inline at write time; dedup runs as a background process. Savings vary by data type: greater for logs and text, lower for already-compressed image formats.
Mechanism 2 Inactive data
Automatic Tiering (FabricPool)
Inactive data cools automatically to Amazon S3. Transparent to applications.
Data unused for X days (configurable) moves from SSD to S3. When accessed again, retrieved transparently with no code changes or manual steps.
Mechanism 3 Experiments
Zero-Copy Clones (FlexClone)
Researchers create instant writable copies of the dataset.
All unchanged blocks are shared with the original. Storage is only consumed for blocks the researcher actually changes. Each data scientist gets an independent copy without duplicating the dataset.
Mechanism 4 Recovery
Point-in-Time Snapshots
Instant rollback safety net for datasets and experiments.
Only changed blocks are stored per snapshot, so storage overhead is minimal. If a data pipeline corrupts a dataset or a FlexClone experiment goes wrong, any prior state can be restored instantly. No performance impact during creation. Applications keep running at full speed.

Backup D: Compute Optimization

Beyond EC2 P5

The Data Science team asked for P5 + SageMaker. Here's how we optimize that ask, and why FSx ONTAP stays the same regardless of which path we choose.

⚠ The Question Nobody Asked the DS Team
Is the most expensive GPU actually solving the right problem?
The requirement is "fast access to specific subsets of 2PB." That is a data access problem, not a computing power problem. H100 GPUs sit idle when storage can't feed them fast enough. Buying more GPU power when the real bottleneck is data access makes training slower per dollar spent.

"Specific subsets" may indicate fine-tuning an existing model rather than training from scratch. Fine-tuning workloads often do not saturate an H100. An older GPU type (A100/P4de) or AWS's own AI chip (Trainium 2) may deliver sufficient results at meaningfully lower cost.
If storage is the bottleneck (GPU is waiting for data)
Fix the data pipeline first. Ensuring fast NFS access and correctly sized hot-tier storage delivers more training throughput per dollar than upgrading the GPU. Choose compute after storage is no longer the bottleneck.
If the GPU is the bottleneck (running at full capacity)
H100 (P5) is justified, but test whether AWS Trainium 2 matches the performance at 30–40% lower cost on your specific model before committing to a full P5 fleet.

Backup E: Honest Alternatives

Why Not FSx for Lustre?

AWS's storage system purpose-built for AI/ML training speed. We evaluated it seriously. Here is why FSx ONTAP still wins for GlobalCorp specifically.

The Alternative
Amazon FSx for Lustre
AWS's purpose-built high-speed storage for AI/ML and scientific computing
Speed ✓ Under 1ms. Built specifically for the random read patterns of AI training. Very fast.
Throughput ✓ Hundreds of GB/second. Scales by adding more storage servers. Best raw speed for ML.
S3 Integration ✓ Can use S3 as its underlying storage. Loads data from S3 automatically on first access.
SageMaker ✓ Works with SageMaker via Lustre protocol (SageMaker mounts the filesystem directly using the Lustre client).
Sync from On-Prem ✗ No built-in NetApp sync. Requires AWS DataSync or custom scripts to transfer from on-prem AFF. DataSync is incremental (only changed files), but adds operational overhead vs. native SnapMirror.
Experiment Copies ✗ No instant copy feature (no FlexClone). Every experiment needs a full data copy. 10 researchers × 500GB = 5TB charged.
Data Compression ✗ No data compression or deduplication. Persistent_2 can release files back to S3, but tiering is not automatic. Effective cost is higher than FSx ONTAP with FabricPool enabled.
Team Familiarity ✗ Completely different tools from NetApp. Our storage team would need to learn Lustre from scratch.
Protocol Support ~ Lustre protocol only. No S3-compatible access for SageMaker jobs, no Windows file sharing (SMB), no direct database access.
Cost ✗ $0.145–$0.29/GB/month. No compression to reduce the bill. No tiering to cheaper storage. Higher effective cost than FSx ONTAP.

Multiple protocols from one system: file sharing + object storage + built-in NetApp sync

When would FSx for Lustre beat FSx ONTAP? New AWS-only workloads with no on-prem NetApp, where data throughput beyond 12.5 GB/second is genuinely required, and the team has no NetApp experience to build on. For GlobalCorp, with existing NetApp on-prem storage, built-in sync as the migration path, and a storage team that knows NetApp, Lustre's speed advantage is outweighed by the cost of building custom data pipelines, losing instant experiment copies (FlexClone), losing data compression savings, and retraining the entire storage team. At this scale, the GPU is the bottleneck, not the storage speed. FSx ONTAP is fast enough.
